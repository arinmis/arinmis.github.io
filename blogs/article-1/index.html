What is data?

2025-01-15

Data is information organized for analysis. It can be numbers, text, images, or anything that computers process. In software engineering, we deal with data constantly — storing it, transforming it, and moving it between systems.

Why it matters

Understanding data structures and databases is fundamental to building scalable systems. Whether you're working with a simple SQLite database or a distributed data lake, the principles are the same: organize data so it can be efficiently stored, queried, and modified.

The challenge isn't just storing data — it's storing it in a way that makes sense for your use case. A social media app needs to query relationships between users quickly. A logging system needs to ingest massive amounts of writes efficiently. A search engine needs to return relevant results in milliseconds.

Types of data

Structured data

Data that follows a fixed schema — rows and columns, like in SQL databases. Think spreadsheets, tables, anything with a rigid format.

Unstructured data

Data without a predefined format — images, videos, PDFs, free-form text. This makes up the vast majority of data in the world. It's powerful but harder to query programmatically.

Semi-structured data

JSON, XML, and similar formats that have structure but no fixed schema. Flexible enough to evolve, structured enough to query.

Data in motion

Data isn't just sitting in databases. It's flowing through systems constantly — API requests, message queues, event streams. Understanding this flow is key to building reliable distributed systems.

Client → API → Message Queue → Worker → Database

Each arrow is a potential failure point. Each step needs proper error handling, retries, and monitoring. This is where things get interesting.

Choosing the right data store

One of the first decisions in any project: where does the data live?

  • Relational (PostgreSQL, MySQL) — when you need transactions, complex joins, and strong consistency
  • NoSQL (MongoDB, Cassandra) — when you need to scale horizontally, handle flexible schemas
  • Key-Value (Redis, DynamoDB) — when you need fast lookups, caching, simple data
  • Search (Elasticsearch) — when you need full-text search, aggregations
  • Time-series (InfluxDB) — when you're tracking metrics, events over time

Code example

Here's a simple data pipeline in Python:

def process_data(items):
    # Filter positive numbers
    positive = [x for x in items if x > 0]
    # Square each value
    squared = [x ** 2 for x in positive]
    return squared

# Example usage
data = [-2, -1, 0, 1, 2, 3]
result = process_data(data)
print(result)  # [1, 4, 9]

The beauty of data processing is that these operations compose. Chain small, focused functions together to build complex pipelines. Map, filter, reduce — the building blocks of data engineering.

Common pitfalls

A few things I've learned the hard way:

  • Over-indexing — adding indexes to every column "just in case" slows down writes significantly
  • Missing null handling — null values sneak in from edge cases you didn't anticipate
  • No backups — it works until it doesn't
  • Ignoring data skew — a few users with millions of records can kill your query performance

Conclusion

Data is the foundation of everything we build. Master the fundamentals — data structures, databases, pipelines — and you'll be equipped to tackle any engineering challenge that comes your way.
